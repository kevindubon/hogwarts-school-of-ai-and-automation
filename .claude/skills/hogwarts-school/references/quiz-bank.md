## Module 1: The Automation Spellbook

### Q1.1: What is the correct order of a basic automation's components?

**A)** Action -> Trigger -> Condition
**B)** Trigger -> Condition -> Action
**C)** Condition -> Action -> Trigger
**D)** Trigger -> Action -> Condition

**Answer:** B
**Explanation:** Trigger (something happens) -> Condition (should we respond?) -> Action (do the thing). It's like the Marauder's Map: the map DETECTS movement (trigger), checks WHO it is (condition), and SHOWS their location (action). Always in that order.

---

### Q1.2: A webhook-based automation fires when:

**A)** A scheduled timer goes off (like an alarm clock)
**B)** An external system sends an HTTP request to a specific URL
**C)** A user manually clicks a "Run" button
**D)** The system polls an API every 5 minutes

**Answer:** B
**Explanation:** Webhooks are like having an owl that's always perched at your window, ready to deliver a message the instant it arrives. No waiting, no polling — just instant delivery when the sending system pushes data to your URL.

### Q1.3: Which of the following is an example of an event-driven automation?

**A)** A report that generates every Monday at 9 AM
**B)** A welcome email sent when a new user signs up
**C)** A script you run manually from the terminal
**D)** A backup that runs every 24 hours

**Answer:** B
**Explanation:** Event-driven means "something happened, so now we react." A new signup is the EVENT; the welcome email is the REACTION. Options A and D are scheduled (time-driven), and C is manual. Even Dobby waited for an event before delivering his warnings.

**Explanation:** Automation excels at repetitive, predictable tasks. ### Q1.9: What does "idempotent" mean in the context of automation?

**A)** The automation runs exactly once and never again
**B)** Running the automation multiple times produces the same result as running it once
**C)** The automation can fix its own errors
**D)** The automation doesn't require any input

**Answer:** B
**Explanation:** Idempotent operations are safe to retry — like casting Lumos twice doesn't make your wand twice as bright. If your automation creates a duplicate record every time it retries, it's NOT idempotent, and you'll end up with 47 copies of the same invoice. Design for idempotency or suffer the consequences.

## Module 2: Potion Ingredients (Data Fundamentals)

### Q2.1: Which of the following is a valid JSON object?

**A)** `{name: "Harry", age: 11}`
**B)** `{"name": "Harry", "age": 11}`
**C)** `{'name': 'Harry', 'age': 11}`
**D)** `("name": "Harry", "age": 11)`

**Answer:** B
**Explanation:** JSON requires double quotes around keys AND string values. Option A is JavaScript (no quotes on keys), option C uses single quotes (forbidden in JSON), and option D uses parentheses (what even is that, a Howler?). JSON is strict — like Snape grading potions essays.

### Q2.4: Which data type would you use to represent whether a candidate has passed a background check?

## Module 3: The Owl Post Network (APIs)

**Answer:** B

---

### Q3.2: Which HTTP method would you use to CREATE a new record?

**A)** GET
**B)** POST
**C)** DELETE
**D)** PATCH

**Answer:** B
**Explanation:** POST creates new things — like enrolling a new student at Hogwarts. GET retrieves existing data (checking the student roster), PUT/PATCH updates records (changing house assignments), and DELETE... well, you get expelled. POST = "I have something NEW to submit."

---

### Q3.3: What does a 200 status code mean?

**A)** The request was redirected to another URL
**B)** The request was successful
**C)** The server encountered an error
**D)** Authentication is required

**Answer:** B
**Explanation:** 200 means "OK — everything worked perfectly." It's the owl returning with a reply letter and a satisfied hoot. The 200-range (200, 201, 204) is the happy zone. If you see a 200, celebrate — your spell worked on the first try, which is more than Neville can say.

### Q3.4: Your API call returns a 404 status code. In Owl Post terms, this means:

### Q3.5: A 500 status code indicates:

**A)** You sent a bad request with missing parameters

**Answer:** C
**Explanation:** 500 = Internal Server Error. This is the Owlery catching fire — it's not YOUR fault, the server broke. There's nothing you can do except wait for the server team to fix it and maybe retry later. 500-range errors are always the server's problem, never yours.

### Q3.6: In a `fetch()` call, where do you typically put the API key for authentication?

It's like the sealed wax stamp on an owl-delivered letter — it proves who you are without exposing your identity to everyone who sees the envelope. Query parameters CAN work but are less secure.

### Q3.9: Which of these is a correct `fetch()` call to get all Harry Potter characters?

**A)** `fetch("hp-api.onrender.com/api/characters")`
**B)** `fetch("https://hp-api.onrender.com/api/characters")`
**C)** `fetch(GET, "https://hp-api.onrender.com/api/characters")`
**D)** `fetch("https://hp-api.onrender.com/api/characters", "GET")`

**Answer:** B
**Explanation:** `fetch()` takes a full URL (including `https://`) as its first argument. GET is the default method, so you don't need to specify it. Option A forgets the protocol, C has the arguments backwards, and D passes the method as a plain string instead of in an options object.

### Q3.10: What is the difference between a query parameter and a path parameter in a URL?

**A)** There is no difference; they're interchangeable
**B)** Path parameters are part of the URL path (`/characters/harry-potter`); query parameters come after `?` (`/characters?house=gryffindor`)
**C)** Query parameters are more secure than path parameters
**D)** Path parameters are only used with POST requests

**Answer:** B
**Explanation:** Path parameters identify a SPECIFIC resource (like calling a student by name), while query parameters FILTER or modify the request (like asking "show me only Gryffindor students"). Think of it as: path = WHO you're looking for, query = HOW you want the results filtered. The Room of Requirement (path) vs. what you need it to contain (query).

### Q7.2: Why is error handling critical in automations?

**Explanation:** An automation without error handling is like a Quidditch match without a referee — when things go wrong (and they WILL), it's pure chaos. Always handle your errors, or they'll handle you.

### Q7.3: What does this error handling pattern do?

```javascript
try {
  const data = await fetch(url);
  const result = await data.json();
} catch (error) {
  console.error("Failed to fetch:", error.message);
}
```

**A)** It prevents the fetch from ever failing
**B)** It attempts the fetch, and if anything goes wrong, it logs the error instead of crashing
**C)** It retries the fetch automatically
**D)** It sends the error to an external monitoring service

**Answer:** B
**Explanation:** Try/catch is your safety net. The `try` block attempts the risky operation; if ANYTHING throws an error, execution jumps to the `catch` block where you handle it gracefully. It's the Cushioning Charm of JavaScript — you'll still fall, but you won't break anything. No retry logic here though; that's an additional enchantment.

## Module 8: Unforgivable Prompts (Prompt Engineering)

### Q8.1: What are the 6 components of effective prompt anatomy?

**A)** Who, What, When, Where, Why, How
**B)** Role, Context, Task, Format, Constraints, Examples
**C)** Input, Process, Output, Error, Retry, Log
**D)** Header, Body, Footer, Sidebar, Caption, Summary

**Answer:** B
**Explanation:** Role (who should Claude be), Context (background information), Task (what to do), Format (how to structure the output), Constraints (what NOT to do), and Examples (show don't tell). It's like the six components of a properly cast spell: intent, incantation, wand movement, focus, power, and practice. Miss one, and the spell fizzles.

---

### Q8.2: The "Imperius Prompt" anti-pattern refers to:

**A)** A prompt that gives Claude too much creative freedom
**B)** A prompt that is so rigidly controlling that Claude can't apply good judgment or ask clarifying questions
**C)** A prompt that asks Claude to do something unethical
**D)** A prompt that's too short to be useful

**Answer:** B
**Explanation:** Like the Imperius Curse removing free will, an Imperius Prompt leaves no room for Claude to exercise judgment. "Do EXACTLY this, ONLY this, NO deviations" sounds precise but often produces worse results because Claude can't flag obvious problems or suggest better approaches. Control is good; over-control backfires.

---

### Q8.3: The "Crucio Prompt" anti-pattern refers to:

**A)** A prompt that inflicts unnecessary complexity — asking for too many things at once in a confusing way
**B)** A prompt that's too simple
**C)** A prompt that uses profanity
**D)** A prompt that takes too long to process

**Answer:** A
**Explanation:** The Crucio Prompt tortures Claude (and your results) with contradictory instructions, nested requirements, unclear priorities, and scope that would make a Ministry bureaucrat weep. "Write a blog post that's also a technical spec that's also a poem, in 200 words, but be thorough." Pain. Pure pain. Break complex requests into clear, sequential steps.

### Q8.4: The "Avada Kedavra Prompt" anti-pattern refers to:

**A)** A prompt that kills the conversation by being so vague or empty that Claude has nothing to work with
**B)** A prompt that deletes data
**C)** A prompt that's too long
**D)** A prompt that crashes Claude

**Answer:** A
**Explanation:** "Do the thing." "Make it better." "Help." These prompts are the Killing Curse for productivity — they destroy any chance of a useful response because there's no context, no specifics, no direction. Claude isn't a mind reader (yet). The vaguer your prompt, the vaguer your response. Give Claude something to work with!

### Q8.9: What role does the "Constraints" component play in a prompt?

### Q8.10: You prompt Claude and the response isn't what you wanted. What's the best next step?
